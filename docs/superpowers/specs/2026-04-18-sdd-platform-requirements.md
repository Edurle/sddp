# 规范驱动开发平台 — 产品需求文档

## 1. 产品概述

规范驱动开发平台是一个面向团队的项目管理工具。核心特色是将开发流程模板化：每个需求必须经过「编写需求→编写规范→编写测试用例」的流程，每步需审核通过，全部通过后拆分为任务，每个任务经过「编写代码→执行测试→完成」的流程。

其中「规范」是给 AI 编程智能体看的结构化文档，规定了数据结构、表设计、页面结构、API 及出入参等约束。平台仅管理规范的编写和流程，不直接集成智能体执行能力。

## 2. 核心实体层级

```
团队 → 项目 → 迭代(Sprint) → 需求 → 任务
  ↕
规范模板(团队级)
```

- 团队是顶级组织单元
- 规范模板作用在团队级别，团队内所有项目共用同一套流程和规范格式
- 需求归属于迭代，迭代归属于项目
- 任务从需求拆分，必须关联一个父需求

## 3. 核心工作流

### 3.1 需求阶段（需求级别）

每一步都需要审核，审核通过后进入下一步且该步骤内容不可修改，驳回需填写理由后退回当前步骤修改。

```
编写需求 → 需求审核 → 编写规范 → 规范审核 → 编写测试用例 → 测试用例审核 → 已通过
    ↑         ↓          ↑          ↓             ↑              ↓
    ←← 驳回 ←←          ←← 驳回 ←←             ←←← 驳回 ←←←
```

1. **编写需求**：填写需求基本信息
2. **编写规范**：按照团队级模板格式编写规范文档
3. **编写测试用例**：编写接口测试用例和端到端测试用例
4. **已通过**：全部审核通过后可拆分任务

### 3.2 任务阶段（任务级别）

```
未开始 → 编写代码 → 执行测试 → 已完成
           ↑         ↓
           ←← 测试失败 ←←
```

1. **未开始**：任务已创建，等待开始编码
2. **编写代码**：根据规范文档编写代码实现
3. **执行测试**：执行关联需求的所有测试用例，记录每个用例的执行结果
4. **完成**：最近一轮测试全部通过后可标记完成

### 3.3 团队级规范模板

团队级规范模板定义了需求阶段「编写规范」步骤产出的文档必须包含的结构化字段：

- **实体定义**：实体名称、字段列表（字段名、类型、约束）
- **数据表设计**：表名、字段、类型、索引、外键关系
- **页面结构**：页面名称、页面编码、元素列表（每个可交互/展示元素赋予唯一编码）、交互行为
- **API 设计**：URL、HTTP 方法、请求参数、响应参数、错误码
- **其他约束**：目录结构、命名规范、编码规范等

### 3.4 测试用例与测试代码对应机制

- 每个测试用例分配唯一编号（如 `TC-{需求编号}-{序号}`）
- 需求阶段编写的测试用例描述测试内容（前置条件、操作步骤、预期结果）
- 任务阶段执行测试时，每条执行记录关联到具体用例编号
- 一个测试用例可被多次执行，每次执行产生独立记录
- 需求维度和迭代维度均可查看测试用例通过率统计

## 4. 功能模块

### 4.1 用户管理

#### 4.1.1 注册账号

- 邮箱 + 密码注册
- 注册后需邮箱验证

#### 4.1.2 平台管理员创建用户

- 平台管理员可直接创建用户账号（设置邮箱、昵称、初始密码）

#### 4.1.3 登录

- 邮箱 + 密码登录
- 支持记住登录状态
- 登录后获取 Token

#### 4.1.4 找回密码

- 通过注册邮箱发送重置链接

#### 4.1.5 修改用户信息

- 修改昵称、头像、密码

#### 4.1.6 用户个人中心

- 查看已加入的团队列表
- 查看待处理事项（待审核、待执行任务等）

#### 4.1.7 平台管理员用户管理

- 查看所有用户列表
- 禁用/启用用户账号

---

### 4.2 团队管理

#### 4.2.1 创建团队

- 设置团队名称、描述
- 创建者自动成为团队所有者

#### 4.2.2 编辑团队

- 修改团队名称、描述

#### 4.2.3 解散团队

- 仅团队所有者可操作
- 需二次确认

#### 4.2.4 邀请成员

- 通过用户名或用户 ID 搜索并发送邀请
- 被邀请者收到邀请通知，可接受或拒绝

#### 4.2.5 分配角色

- 为团队成员分配角色（角色在权限管理模块中自定义创建）

#### 4.2.6 移出成员

- 将成员移出团队
- 团队所有者不可被移出

#### 4.2.7 成员列表

- 查看团队成员、角色、加入时间
- 支持按角色筛选

#### 4.2.8 转让团队

- 团队所有者可将所有权转让给其他成员

---

### 4.3 项目管理

#### 4.3.1 创建项目

- 在团队下创建项目
- 设置项目名称、描述、开始日期

#### 4.3.2 编辑项目

- 修改项目名称、描述、开始日期

#### 4.3.3 归档项目

- 手动归档，归档后项目只读
- 不受截止日期影响自动归档

#### 4.3.4 删除项目

- 仅无活跃迭代时可删除
- 需二次确认

#### 4.3.5 项目列表

- 展示团队下所有项目
- 支持按状态（活跃/归档）筛选

#### 4.3.6 项目详情

- 展示项目基本信息
- 展示当前活跃迭代的进度统计（需求数、任务数、测试通过率）

#### 4.3.7 项目起止日期

- 开始日期手动设置
- 结束日期自动取最新迭代的截止日期

---

### 4.4 迭代管理

#### 4.4.1 创建迭代

- 在项目下创建迭代（Sprint）
- 设置迭代名称、目标描述、开始日期、截止日期

#### 4.4.2 编辑迭代

- 修改迭代名称、目标、截止日期
- 开始日期一旦设定不可修改

#### 4.4.3 启动迭代

- 将迭代状态从「计划中」变更为「进行中」

#### 4.4.4 完成迭代

- 手动点击完成
- 系统验证：
  - 迭代内所有需求已拆分任务
  - 所有任务状态为「已完成」
  - 最近一次测试执行记录中，所有测试用例全部通过
  - 以上条件任一不满足则提示具体原因，不允许完成

#### 4.4.5 迭代列表

- 展示项目下所有迭代
- 支持按状态（计划中/进行中/已完成）筛选

#### 4.4.6 迭代看板

- 以看板视图展示迭代内需求状态流转

#### 4.4.7 迭代统计

- 需求完成数/总数
- 任务完成数/总数
- 测试用例通过率

#### 4.4.8 需求优先级

- 需求使用数字优先级，数字越大优先级越高

---

### 4.5 需求管理

#### 4.5.1 创建需求

在迭代下创建需求，通用字段：标题、优先级（数字，越大越高）、类型（功能/优化/缺陷）。

按类型需填写不同内容：
- **功能**：功能描述（描述要实现的新功能）
- **优化**：功能变更描述（描述现有功能的变更内容）
- **缺陷**：缺陷描述（复现步骤、预期行为、实际行为）+ 修改后的新功能描述

#### 4.5.2 编辑需求

- 修改标题、描述、优先级、类型
- 仅在「编写需求」步骤时可编辑

#### 4.5.3 删除需求

- 仅在「编写需求」步骤时可删除

#### 4.5.4 需求列表

- 展示迭代下所有需求
- 支持按状态、优先级、类型筛选和排序

#### 4.5.5 需求详情

- 展示需求信息
- 展示当前流程步骤
- 展示关联的规范文档、测试用例、任务列表、审核记录

#### 4.5.6 需求流程步骤

每步提交审核时指定审核人，审核人可查看当前步骤的产出物。通过后进入下一步且内容不可修改，驳回需填写理由后退回当前步骤。

1. **编写需求** → 提交审核 → 审核通过后不可修改
2. **编写规范** → 提交审核 → 审核通过后不可修改
3. **编写测试用例** → 提交审核 → 审核通过后不可修改
4. **已通过** → 可拆分任务

---

### 4.6 规范管理

#### 4.6.1 规范层级

- **团队级模板**：定义规范文档的格式和必填结构（团队级别，固定）
- **需求级文档**：需求阶段「编写规范」步骤产出的实际规范内容，按模板格式填写

#### 4.6.2 查看规范文档

- 在需求详情中查看该需求的规范文档

#### 4.6.3 编辑规范文档

- 在需求处于「编写规范」步骤时可编辑
- 审核驳回后退回修改时也可编辑

#### 4.6.4 规范文档内容

遵循团队级模板定义的结构，包含：
- 实体定义（名称、字段列表、字段类型、约束）
- 数据表设计（表名、字段、类型、索引、外键关系）
- 页面结构（页面名称、页面编码、元素列表含唯一编码、交互行为）
- API 设计（URL、HTTP 方法、请求参数、响应参数、错误码）
- 其他约束（目录结构、命名规范、编码规范等）

#### 4.6.5 规范版本记录

- 每次保存产生一个版本记录
- 可查看历史版本

---

### 4.7 任务管理

#### 4.7.1 创建任务

- 从已通过的需求拆分任务
- 设置任务标题、描述、指派人

#### 4.7.2 编辑任务

- 修改任务标题、描述、指派人
- 仅在「编写代码」步骤时可编辑

#### 4.7.3 删除任务

- 仅在「编写代码」步骤时可删除

#### 4.7.4 任务列表

- 展示需求下的所有任务
- 支持按状态、指派人筛选

#### 4.7.5 任务详情

- 展示任务信息
- 展示关联的需求规范文档
- 展示当前流程步骤

#### 4.7.6 任务流程步骤

1. **编写代码**：根据规范文档编写代码实现 → 完成后进入下一步
2. **执行测试**：执行关联需求的所有测试用例，记录每个用例的执行结果
3. **完成**：最近一轮测试全部通过后可标记完成；如有失败用例则退回「编写代码」步骤

#### 4.7.7 任务与测试用例的关联

- 任务继承其父需求的所有测试用例
- 每次执行测试产生一轮测试记录，关联到具体任务
- 同一用例可多次执行，每次产生独立记录
- 只有最近一轮测试全部通过才能完成任务

---

### 4.8 测试管理

#### 4.8.1 测试用例管理

- 在需求阶段「编写测试用例」步骤中创建
- 每个用例包含：
  - 用例编号（自动生成，如 `TC-{需求编号}-{序号}`）
  - 用例标题
  - 类型（接口测试/端到端测试）
  - 前置条件
  - 操作步骤
  - 预期结果
  - 关联的 API 路径或页面元素编码
- 支持按需求、按类型筛选查看
- 审核通过后不可修改（随需求锁定）

#### 4.8.2 测试执行记录

- 在任务「执行测试」步骤中创建
- 每轮执行产生一组记录，关联到具体任务
- 每条记录包含：
  - 用例编号
  - 执行状态（通过/失败/跳过）
  - 实际结果
  - 失败原因（失败时必填）
  - 执行时间
- 支持按任务、按用例、按状态筛选查看

#### 4.8.3 测试统计

- **需求维度**：该需求下所有测试用例的最近执行结果统计
- **迭代维度**：迭代内所有需求的测试用例通过率
- **项目维度**：项目下所有迭代的测试通过率趋势

---

### 4.9 权限管理

#### 4.9.1 角色定义

- 团队级自定义角色，每个角色有一个权限项集合
- 系统内置角色（不可删除不可修改）：
  - **团队所有者**：拥有所有权限
  - **团队管理员**：除解散团队、转让团队外的所有权限

#### 4.9.2 权限项

- 项目管理（创建、编辑、归档、删除）
- 迭代管理（创建、编辑、启动、完成）
- 需求管理（创建、编辑、删除、审核需求、审核规范、审核测试用例）
- 任务管理（创建、编辑、删除、执行测试、完成任务）
- 成员管理（邀请、移出、分配角色）
- 规范模板管理（编辑团队级模板）

#### 4.9.3 角色分配

- 团队所有者和管理员可为成员分配角色
- 一个成员可有多个角色，权限取并集
- 团队所有者角色不可被移除，始终至少有一名所有者

#### 4.9.4 平台管理员

- 系统级角色，不属于团队角色体系
- 可查看所有用户、创建用户、禁用/启用用户账号
- 可查看所有团队列表（不可操作团队内部数据）
