## 已发现问题

> 实测过程中发现的问题，待后续统一修复。

### ISSUE-001: 需求进度条状态不全

- **发现阶段**: Task 9（测试用例管理）完成后验证时
- **现象**: 需求详情页的步骤条只显示 4 个状态（需求、规范、测试、已通过），缺少 7 个完整状态（drafting_req → reviewing_req → drafting_spec → reviewing_spec → drafting_tests → reviewing_tests → approved）
- **期望**: 步骤条应展示完整 7 阶段生命周期，区分「编写」和「审核」子阶段
- **影响页面**: `RequirementDetailPage.vue` 中的 `req-detail-step-nav` 步骤条组件

### ISSUE-005: 禁用用户仍可登录

- **发现阶段**: Task 15（管理后台）
- **现象**: 管理员通过 admin API 将用户设为 `is_active=false` 后，该用户仍能成功登录并获取 token
- **期望**: 被禁用的用户登录时应返回错误（如"账号已被禁用"），或登录成功但所有请求被拦截
- **影响范围**: 登录接口 `POST /api/v1/auth/login` 未检查 `is_active` 状态

### ISSUE-004: 测试执行记录不显示失败原因

- **发现阶段**: Task 11（测试执行）
- **现象**: 测试执行结果为「失败」时，前端测试记录表格中看不到失败原因（failure_reason）字段，无法快速了解失败详情
- **期望**: 失败状态的测试记录应在列表中直接展示失败原因，或提供展开/弹窗查看详情的方式
- **影响页面**: `TaskDetailPage.vue` 测试执行 tab 的记录表格

### ISSUE-003: 任务详情页不显示 Git 分支信息

- **发现阶段**: Task 10（任务管理）
- **现象**: 通过 API 成功设置了 git_branch、commit_sha、pr_url，但任务详情页前端不显示这些 Git 信息
- **期望**: 任务详情页应展示 Git 分支、Commit SHA、PR URL 等信息
- **影响页面**: `TaskDetailPage.vue` 侧边栏或详情区域

### ISSUE-002: 测试用例查看不便

- **发现阶段**: Task 9（测试用例管理）
- **现象**: 测试用例列表中只能看到标题等基本信息，必须点击「编辑」按钮才能查看完整详情（步骤、预期结果、前置条件等）
- **期望**: 应提供「查看详情」操作（点击行展开/抽屉/弹窗），方便快速查看用例完整内容，不需要进入编辑模式
- **影响页面**: `RequirementDetailPage.vue` 中测试用例 tab 的表格/列表区域

## Task 8: 阶段七 — 规范文档（Spec）

### 8.1 编写规范

- [ ] **Step 1: 打开规范编辑器**

- [ ] **Step 2: 填写实体定义 — User**

在实体定义区域：
- 描述: `博客系统用户实体`
- 字段:
  - `id` (integer, 主键, 自增)
  - `email` (string, 唯一, 非空)
  - `nickname` (string, 非空, 最大32字符)
  - `password_hash` (string, 非空)
  - `avatar` (string, 可空)
  - `is_active` (boolean, 默认true)
  - `created_at` (datetime, 自动)
  - `updated_at` (datetime, 自动)

- [ ] **Step 3: 填写数据表设计**

- 表名 `users`，描述 `用户表`
- 字段: id(INT PK AUTO), email(VARCHAR(255) UNIQUE NOT NULL), nickname(VARCHAR(32) NOT NULL), password_hash(VARCHAR(255) NOT NULL), avatar(VARCHAR(512)), is_active(BOOLEAN DEFAULT TRUE), created_at(TIMESTAMP), updated_at(TIMESTAMP)
- 索引: idx_users_email (UNIQUE on email), idx_users_nickname (on nickname)

- [ ] **Step 4: 填写页面结构**

- 页面 `登录页`，代码 `LoginPage`，路由 `/login`
  - 元素: 邮箱输入框, 密码输入框, 登录按钮, 注册链接
  - 交互: 点击登录 → POST /api/auth/login → 跳转 dashboard
- 页面 `注册页`，代码 `RegisterPage`，路由 `/register`
  - 元素: 邮箱输入框, 昵称输入框, 密码输入框, 确认密码输入框, 注册按钮
  - 交互: 点击注册 → POST /api/auth/register → 跳转登录页

- [ ] **Step 5: 填写 API 设计**

- `POST /api/v1/auth/register` — 注册
  - 请求: email, password, nickname
  - 响应 201: { code: 0, data: { id, email, nickname } }
  - 错误: 40002 邮箱已注册, 40001 参数校验失败
- `POST /api/v1/auth/login` — 登录
  - 请求: email, password
  - 响应 200: { code: 0, data: { token, user } }
  - 错误: 40003 密码错误, 40004 邮箱未验证

- [ ] **Step 6: 填写其他约束**

- 目录结构: `src/views/auth/`, `src/api/auth.ts`, `backend/app/api/auth.py`
- 命名规范: RESTful API, 组件 PascalCase, 文件 kebab-case
